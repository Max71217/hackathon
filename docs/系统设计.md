# 系统设计与技术选型

## 1. 整体架构设计
本系统采用了高度解耦的模块化设计，整体架构自下而上分为四层：感知层、引擎层、逻辑层与应用层。

### 1.1 感知层 (Perception Layer)
*   **组件**：`SmartPdfParser`
*   **职责**：负责教材的加载与预处理。使用 PyMuPDF 对 PDF 进行极速逐页解析，根据提取的字符数量，将页面数据转化为“纯文本（Text）”或“图像 Base64 流（Image）”。

### 1.2 引擎层 (AI Engine Layer)
*   **组件**：智谱大模型双引擎 (`glm-4-flash` & `glm-4.6v`)
*   **职责**：针对感知层分流的数据进行结构化提取。
    *   **视觉大脑**：`glm-4.6v` 专攻医学插图与扫描件，具有极强的学术图片理解力，规避了常规模型的异常安全拦截。
    *   **文本大脑**：`glm-4-flash` 以极低延迟完成海量纯文本页面的知识点挖掘。
*   **容错机制**：内置多级异常捕获与正则表达式 JSON 提纯器，确保单页提取失败自动降级跳过，系统永不宕机。

### 1.3 逻辑层 (Logic Layer)
*   **组件**：DeepSeek 融合中心 (`deepseek-v4-pro`)
*   **职责**：承担全系统复杂度最高的跨教材图谱对齐任务（Ontology Alignment）。利用 DeepSeek 原生支持的 `json_object` 格式与强悍的逻辑推理能力，一次性读入多个单本图谱，通过深层语义比对，输出去重、连贯且高压缩比的全局 Knowledge Graph。

### 1.4 应用层 (Application Layer)
*   **组件**：Streamlit Web UI + ECharts
*   **职责**：提供用户交互入口。左侧管理多源文件上传流，中央通过 ECharts 动态渲染力导向知识图谱，右侧集成多图谱切换与 Graph-RAG 智能问答面板。

---

## 2. 核心数据流转 (Data Flow)

1. **输入阶段**：用户上传 PDF/TXT -> 存入临时目录。
2. **提取阶段**：
   * 页面字符 > 400 -> 路由至智谱 `glm-4-flash`。
   * 页面字符 <= 400 -> 转换为图片 -> 路由至智谱 `glm-4.6v`。
   * **输出**：严格符合 Pydantic Schema 的单本图谱 JSON，存入 `st.session_state` 字典。
3. **融合阶段**：用户下发指令 -> 提取 `session_state` 中的多本图谱 -> 并行输入 DeepSeek-V4-Pro -> 脑内比对去重 -> **输出**：跨书整合版图谱。
4. **服务阶段**：用户提问 -> 后端拼装图谱拓扑结构（实体+关系）作为上下文 -> 大模型生成带有医学来源标注的回答。

## 3. 技术栈选型依据

| 技术领域 | 选型 | 决策依据 |
| :--- | :--- | :--- |
| **前端框架** | Streamlit | 黑客松敏捷开发利器，纯 Python 构建无缝对接 AI 逻辑，内置 `session_state` 完美解决图谱状态管理。 |
| **可视化图表** | streamlit-echarts | 相比于静态网络图库（如 NetworkX），ECharts 力导向图支持拖拽、缩放、节点高亮等高级交互特性，业务展示效果极佳。 |
| **文档解析** | PyMuPDF (fitz) | C 语言底层，解析 PDF 文本及渲染图像的速度比 PDFMiner 等纯 Python 库快数十倍。 |
| **感知大模型** | 智谱 AI 系列 | `glm-4.6v` 具备国内顶级的学术图表多模态理解能力，且安全审核策略对正规医学解剖图谱包容度高，避免了因“拦截”导致的流程中断。 |
| **逻辑大模型** | DeepSeek-V4-Pro | 逻辑推理与 JSON 代码生成能力断层领先，超长上下文可完美承载海量图谱节点的聚合与比对分析。 |
| **数据契约** | Pydantic | 强制类型校验，防止大模型幻觉字段污染系统流转，降低 Debug 成本。 |
